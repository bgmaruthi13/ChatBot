# PDF ChatBot

Upload a PDF, wait a few seconds while it's processed, then ask it
questions and get answers with page citations. Runs fully locally with
zero API keys — a real generative LLM is optional, not required.

## How it works

```
Upload → extract text (pypdf) → chunk (~800 words, overlap 150)
       → embed (sentence-transformers, local, CPU) → store (Chroma, on disk)

Ask a question → embed the question → similarity search in Chroma
               → answer generated by the configured LLM provider
                 (default: ExtractiveProvider, returns the top-matching
                 passages verbatim with page citations - no API key
                 needed; see "Enabling a real LLM" below to swap in
                 Claude/OpenAI/Ollama)
```

Everything runs in a plain Django app - no Celery/Redis (a background
`threading.Thread` handles processing so uploads don't block), no
separate frontend build (server-rendered templates + a little vanilla
JS for status polling). See [BACKLOG.md](BACKLOG.md) for the full
epic/story breakdown and [PROGRESS.md](PROGRESS.md) for build status.

## Setup

```bash
# from the ChatBot/ directory
python -m venv venv
venv\Scripts\activate          # Windows
# source venv/bin/activate     # macOS/Linux

pip install -r requirements.txt

copy .env.example .env         # Windows
# cp .env.example .env         # macOS/Linux

python manage.py migrate
python manage.py runserver
```

Then open http://127.0.0.1:8000/ - upload a PDF under **Documents**,
wait for its status to reach **Ready**, then start a chat from
**Chat**.

### Or with Docker

```bash
cp .env.example .env   # then edit if needed
docker compose up --build
```

Serves the same app at http://127.0.0.1:8000/. `db.sqlite3`, `media/`,
and `chroma_data/` persist on the host via a bind mount, so data
survives container rebuilds. (Not verified against a real Docker
install in this environment - Docker wasn't available here - but the
Dockerfile/compose file follow standard, unremarkable patterns.)

The first upload will download a small (~90MB) embedding model from
Hugging Face on first use; after that it's cached locally and every
subsequent run is offline.

### Running tests

```bash
python manage.py test
```

## Enabling a real LLM

By default, chat answers come from `ExtractiveProvider` - it returns
the most relevant passages verbatim with page citations, no API key
required. To get real generated answers instead, set two things in
`.env` and restart the server - no code changes:

| Provider | `.env` settings |
| --- | --- |
| Claude (Anthropic) | `LLM_PROVIDER=claude`, `ANTHROPIC_API_KEY=...` (optionally `ANTHROPIC_MODEL=...`) |
| OpenAI | `LLM_PROVIDER=openai`, `OPENAI_API_KEY=...` (optionally `OPENAI_MODEL=...`) |
| Ollama (local, offline) | `LLM_PROVIDER=ollama` and a running [Ollama](https://ollama.com) server (optionally `OLLAMA_BASE_URL`, `OLLAMA_MODEL`) |

If `LLM_PROVIDER` is unset, misspelled, or its required key is missing,
the app logs a warning and falls back to `ExtractiveProvider` rather
than erroring - chat always works.

## Project layout

```
config/            Django project settings, root urls
documents/          Upload, extraction, chunking, embedding orchestration
  services.py         extract_text(), chunk_text(), process_document()
  tests/               automated tests + PDF fixtures
chat/                Retrieval + chat UI
  services.py            retrieve(), ask()
  llm/                    LLMProvider interface + extractive/claude/openai/ollama
vectorstore/          Embedding model + Chroma client wrappers
static/css/theme.css   Azure Portal-inspired design tokens
templates/               Server-rendered UI (base shell, documents, chat)
```

## Known limitations

- **No OCR** - only digital-text PDFs are supported. Scanned/image-only
  PDFs extract as empty and fail processing. OCR support (Tesseract +
  Poppler) was attempted and dropped - see BACKLOG.md Epic 2, story 2.2
  for why, and how to reinstate it if you have those binaries installed.
- **50MB / 500-page limits** on uploads, enforced server-side.
- **Single-user, no auth** - anyone with access to the app can see all
  uploaded documents.
